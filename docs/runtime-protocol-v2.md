# Runtime Protocol v2

## Overview

This document defines a **project-owned runtime contract** for the Pi WebUI/runtime migration. It is inspired by concepts from AI SDK but is **not** a claim of full AI SDK compatibility. This protocol is designed specifically for `pi-matrix-agent`'s server-authoritative architecture.

**Status**: Draft foundation for migration. Current code does **not** yet fully match this contract.

---

## Design Principles

### Server-Authoritative State

The server is the **source of truth** for all session state:

1. **Session lifecycle**: Server creates/manages/destroys sessions
2. **Tool execution**: All tools run on the server with proper permissions
3. **Model/provider selection**: Configuration lives on the server
4. **Transcript persistence**: All conversation history is server-side
5. **Live event stream**: Server owns the inference loop and emits events

**The browser never owns sessions, config, providers, or state.**

### Canonical Shapes

This protocol defines canonical shapes for:

- Runtime messages (persisted + live)
- Message parts (text, reasoning, tool-call, tool-result)
- Stream events (snapshot, message lifecycle, tool lifecycle, turn lifecycle)
- Capability/state types (interrupt, stop, etc.)

---

## Canonical Runtime Message Model

A **runtime message** represents a single turn of communication.

```typescript
interface RuntimeMessage {
  id: string;           // Unique message ID (server-generated)
  role: "user" | "assistant" | "system";
  parts: RuntimeMessagePart[];  // Ordered parts
  createdAt: string;    // ISO 8601 timestamp
  metadata?: {          // Optional metadata
    [key: string]: unknown;
  };
}
```

### Message IDs

- **Server-generated**: Message IDs are generated by the server
- **Stable**: Once assigned, IDs never change
- **Unique**: IDs are unique within a session

### Roles

| Role | Description |
|------|-------------|
| `user` | User prompt/input |
| `assistant` | Assistant response (includes text, reasoning, tool calls) |
| `system` | System/instruction messages |

---

## Canonical Message Part Model

A **message part** represents a single unit of content within a message.

```typescript
type RuntimeMessagePart =
  | TextPart
  | ReasoningPart
  | ToolCallPart
  | ToolResultPart;

interface TextPart {
  type: "text";
  text: string;
}

interface ReasoningPart {
  type: "reasoning";
  text: string;  // Thinking/reasoning content
}

interface ToolCallPart {
  type: "tool-call";
  toolCallId: string;  // Server-generated tool call ID
  name: string;        // Tool name
  arguments: unknown;  // Tool arguments (structured object)
}

interface ToolResultPart {
  type: "tool-result";
  toolCallId: string;  // References the tool call
  name: string;        // Tool name
  result: string;      // Result text
  isError?: boolean;   // Whether this is an error result
}
```

### Part Ordering

Parts are **ordered** within a message:

```
User message: [TextPart]

Assistant message: [TextPart, ToolCallPart, ToolCallPart, TextPart]
                   ^        ^          ^          ^
                   |        |          |          +-- Follow-up text
                   |        |          +------------- Second tool call
                   |        +-------------- First tool call
                   +--------------- Opening text
```

### Tool Call/Result Pairing

- Tool calls are parts of the **assistant message** that initiated them
- Tool results are parts within messages, not separate messages with role "tool"
- The `toolCallId` links calls to results
- The canonical runtime v2 model uses only roles: `"user" | "assistant" | "system"`
- Tool calls and tool results are represented as **parts**, not as separate "tool" messages

## Canonical Stream Event Model

The SSE stream emits **runtime events** that describe state changes.

### Event Categories

#### 1. Snapshot/Init Event

Emitted when SSE connection opens, providing initial state.

```typescript
interface SnapshotEvent {
  type: "snapshot";
  sessionId: string;
  messages: RuntimeMessage[];  // Historical messages
  capabilities: Capabilities;  // Available capabilities
  state: ConnectionState;      // Current connection state
}
```

#### 2. Message Lifecycle Events

```typescript
// Message begins (new message appended)
interface MessageStartEvent {
  type: "message-start";
  messageId: string;
  role: "user" | "assistant" | "system";
}

// Message updated (part appended/modified)
interface MessageUpdateEvent {
  type: "message-update";
  messageId: string;
  part: RuntimeMessagePart;  // New or updated part
  partial?: boolean;          // true if content is incomplete
}

// Message finalized (complete, no more updates)
interface MessageFinalizeEvent {
  type: "message-finalize";
  messageId: string;
}
```

#### 3. Tool Lifecycle Events

```typescript
// Tool call begins
interface ToolStartEvent {
  type: "tool-start";
  toolCallId: string;
  messageId: string;    // Parent message
  name: string;
  arguments: unknown;
}

// Tool call updated (progress/streaming)
interface ToolUpdateEvent {
  type: "tool-update";
  toolCallId: string;
  progress?: string;    // Progress indicator
}

// Tool call ends (result available)
interface ToolEndEvent {
  type: "tool-end";
  toolCallId: string;
  result: string;
  isError?: boolean;
}
```

#### 4. Turn/Run Lifecycle Events

```typescript
// Turn begins
interface TurnStartEvent {
  type: "turn-start";
  turnId: string;
  sessionId: string;
}

// Turn ends
interface TurnEndEvent {
  type: "turn-end";
  turnId: string;
  success: boolean;
  error?: string;
}
```

#### 5. State/Capability Events

```typescript
// Connection state change
interface StateChangeEvent {
  type: "state-change";
  state: ConnectionState;
}

// Capabilities updated
interface CapabilitiesEvent {
  type: "capabilities";
  capabilities: Capabilities;
}
```

#### 6. Error Event

```typescript
interface ErrorEvent {
  type: "error";
  message: string;
  code?: string;
  turnId?: string;    // Associated turn if applicable
}
```

### Event Union

```typescript
type RuntimeEvent =
  | SnapshotEvent
  | MessageStartEvent
  | MessageUpdateEvent
  | MessageFinalizeEvent
  | ToolStartEvent
  | ToolUpdateEvent
  | ToolEndEvent
  | TurnStartEvent
  | TurnEndEvent
  | StateChangeEvent
  | CapabilitiesEvent
  | ErrorEvent;
```

## Persisted vs. Live-Only

### Persisted (in session file)

Stored in the JSONL session file:

- `RuntimeMessage` (complete messages)
- Message parts with final content
- Turn boundaries (implicitly via message ordering)

### Live-Only (streamed, not persisted)

Streamed via SSE but not persisted:

- Partial/incomplete content (streaming deltas)
- Tool progress updates
- Connection state changes
- Capability changes
- Turn lifecycle events (turn-start, turn-end)

**Note**: The current implementation persists turns differently. This is a target state.

---

## Reload/Replay Behavior

### On Page Load

1. Fetch historical transcript via `GET /api/live/rooms/:roomKey/transcript`
2. Convert transcript items to `RuntimeMessage[]`
3. Connect to SSE stream for live updates
4. SSE emits `snapshot` event with current state

### On Reload

1. Clear client state
2. Re-fetch transcript
3. Reconnect to SSE
4. Replay from current server state

**No client-side persistence.** The server is always the source of truth.

## Capabilities

Capabilities describe what the runtime supports:

```typescript
interface Capabilities {
  interrupt?: boolean;    // Can interrupt ongoing turns?
  stop?: boolean;         // Can stop ongoing turns?
  streaming?: boolean;    // Does streaming work?
  toolCalls?: boolean;    // Are tool calls supported?
  reasoning?: boolean;    // Is reasoning/thinking supported?
}
```

### Interrupt

**Interrupt is a separate capability and is NOT assumed.**

- Not part of the base protocol
- Requires explicit capability advertisement
- Requires additional UI controls (stop button)
- Not implemented in Milestone 1

## Migration Strategy

### Current State (As-Is)

| Component | Current Implementation |
|-----------|----------------------|
| Message model | `TranscriptItem[]` (flat list) |
| Parts | Flat kinds: `user_message`, `assistant_message`, `tool_start`, etc. |
| Events | WebUI events: `turn_start`, `message_update`, `tool_start`, etc. |
| turnId | Generated client-side in emitter, provided via SSE |
| POST response | Returns `{accepted, roomKey, roomId, sessionId, timestamp}` (no turnId) |

### Target State (To-Be)

| Component | Target Implementation |
|-----------|----------------------|
| Message model | `RuntimeMessage[]` with nested parts |
| Parts | Discriminated union: `text`, `reasoning`, `tool-call`, `tool-result` |
| Events | Runtime events: `snapshot`, `message-start`, `message-update`, etc. |
| turnId | Server-generated, provided via SSE |
| POST response | Returns `{accepted, roomKey, roomId, sessionId, timestamp}` (no turnId) |

### Migration Phases

**Milestone 1 (This PR)**:
- Add protocol design doc (`runtime-protocol-v2.md`)
- Add server-side protocol types (`src/runtime-protocol.ts`)
- Add frontend protocol types (`frontend/assistant-ui-spike/src/runtime-protocol.ts`)
- Fix contract drift (turnId not in POST response)
- Add tests for protocol types

**Milestone 2 (Future)**:
- Add snapshot event to SSE stream
- Add message-start/update/finalize events
- Update transcript to return message-shaped data
- Update adapter to consume new event model

**Milestone 3 (Future)**:
- Full behavioral migration
- Add interrupt capability
- Add reload/replay from snapshot

## Current vs. Target: Gap Analysis

### What Matches

- ✅ Server-authoritative architecture
- ✅ SSE-based event streaming
- ✅ Non-blocking prompt submission
- ✅ turnId provided via SSE (not POST response)

### What Does Not Match Yet

- ❌ Transcript returns flat `TranscriptItem[]`, not `RuntimeMessage[]`
- ❌ Events use old WebUI schema, not runtime event schema
- ❌ No snapshot event on connection
- ❌ No message-start/update/finalize events
- ❌ Parts are flat kinds, not nested discriminated union
- ❌ No capabilities advertisement
- ❌ No interrupt capability

### Notes

This protocol is **additive**. The current implementation continues to work while the new protocol is layered on top. Migration happens incrementally.

## Design Decisions

### Why Not Full AI SDK Compatibility?

The AI SDK runtime model assumes:
- Client-side state management
- Interrupt/stop as first-class capabilities
- Different event semantics

Pi-matrix-agent has:
- Server-authoritative state
- Matrix integration
- Different tool model
- Existing user base

This protocol borrows useful concepts (message parts, streaming events) while respecting the existing architecture.

### Why Nested Parts?

Nested parts allow:
- Clear separation of content types
- Easy extension (new part types)
- Better alignment with reasoning models
- Cleaner tool call/result pairing

### Why Server-Generated IDs?

Server-generated IDs ensure:
- Consistency across reloads
- No client-side ID conflicts
- Server remains authoritative

## Appendix: Event Sequence Example

### Complete Turn Sequence

```
[Client connects]
  → snapshot
    { type: "snapshot", sessionId: "s1", messages: [...], capabilities: {...} }

[User submits prompt]
  → turn-start
    { type: "turn-start", turnId: "t1", sessionId: "s1" }
  → message-start
    { type: "message-start", messageId: "m1", role: "user" }
  → message-update
    { type: "message-update", messageId: "m1", part: { type: "text", text: "Hello" } }
  → message-finalize
    { type: "message-finalize", messageId: "m1" }

[Assistant responds]
  → message-start
    { type: "message-start", messageId: "m2", role: "assistant" }
  → message-update
    { type: "message-update", messageId: "m2", part: { type: "text", text: "Hi!" }, partial: true }
  → message-update
    { type: "message-update", messageId: "m2", part: { type: "text", text: " Hi!" } }
  → tool-start
    { type: "tool-start", toolCallId: "tc1", messageId: "m2", name: "bash", arguments: {...} }
  → tool-end
    { type: "tool-end", toolCallId: "tc1", result: "output", isError: false }
  → message-update
    { type: "message-update", messageId: "m2", part: { type: "text", text: " Done." } }
  → message-finalize
    { type: "message-finalize", messageId: "m2" }
  → turn-end
    { type: "turn-end", turnId: "t1", success: true }
```
